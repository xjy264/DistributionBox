# v1.21 完成记录（Phase 2：抢修同构 task-item）

## 做了什么
- 按 v1.20/巡检 Phase1 方案，将“抢修”改为主单+子单模型。
- 新增迁移 `V26__create_sys_emergency_repair_task_item.sql`：
  - `sys_emergency_repair_task`
  - `sys_emergency_repair_item`
  - 约束：`UNIQUE(task_id, box_id)`、`task_id` 外键到 task、`box_id` 外键到 `sys_box`
- 后端新增 repair-task / repair-item 分层代码与接口。
- 前端 `Fix.vue` 改用新 task 接口；`BoxDetail.vue` 抢修记录改用 repair-item 明细历史。
- 旧 `/emergency-repair/**` 保留，未删除。

## 改了什么
### 数据库
- `sql/migrations/V26__create_sys_emergency_repair_task_item.sql`

### 后端新增
- Entity：`RepairTask`、`RepairItem`
- Mapper：`RepairTaskMapper`、`RepairItemMapper`
- Service：`IRepairTaskService`、`IRepairItemService`
- ServiceImpl：`RepairTaskServiceImpl`、`RepairItemServiceImpl`
- DTO：`RepairTaskSaveDto`、`RepairTaskViewDto`、`RepairItemViewDto`
- Controller：`RepairTaskController`、`RepairItemController`

### 前端修改
- `frontend/src/views/Fix.vue`
- `frontend/src/views/BoxDetail.vue`

### 文档
- `docs/demands/v1.21.md`

## 为什么这样改
- 与巡检同构后，数据语义一致（任务视角 + 设备明细视角）。
- 任务层可汇总，配电箱层可追溯历史，满足 v1.20/v1.21 目标。
- 保留旧接口，降低回归风险，便于平滑迁移。

## 验证步骤
1. 执行迁移到 V26。
2. 在“抢修”页面新增任务，选择多个配电箱并保存。
3. 验证库表：主表 1 条、子表 N 条；同 task 下重复 box 受唯一约束限制。
4. 打开某配电箱详情页，在“抢修记录”看到该箱抢修明细历史（repair-item/page）。
5. 编辑/删除任务验证任务与明细联动。
6. 执行后端编译与前端 build 验证通过。
