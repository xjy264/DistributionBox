# v1.14 完成记录

## 现象复现
- 登录页输入账号密码后，页面又回到登录页。
- 浏览器控制台可见 `/api/location/tree`、`/api/box/page` 返回 401。

## 根因
这不是纯前端路由问题，而是“登录后业务接口 401 导致前端拦截器强制登出并跳回 `/login`”的连锁反应。

链路如下：
1. `/login` 成功
2. 跳转 `/box`
3. `/box` mounted 请求接口返回 401
4. `http.ts` 的 401 处理触发 `store.logout()` + `window.location.href='/login'`
5. 用户看起来像“点登录又回登录页”

另外，组件外（router/http）使用 Pinia 未显式绑定同一个 pinia 实例，也增加了登录态读取不稳定风险。

## 改动内容

### 1) 固化 Pinia 单例，避免组件外状态读取不一致
- 新增：`frontend/src/stores/index.ts`
- 修改：
  - `frontend/src/main.ts` 使用同一个 `pinia` 实例
  - `frontend/src/api/http.ts` 在拦截器里改为 `useUserStore(pinia)`
  - `frontend/src/router/index.ts` 守卫中改为 `useUserStore(pinia)`

### 2) 简化路由守卫，去掉会导致阻塞/回退的分支
- `frontend/src/router/index.ts`
- 逻辑：未登录统一跳 `/login`；已登录访问 `/login` 时跳 `/box`。

### 3) 后端鉴权临时放开（用于恢复可用性，避免登录后 401 回跳）
- `backend/src/main/java/com/distributionbox/security/SecurityConfig.java`
- 将 `anyRequest().authenticated()` 调整为 `anyRequest().permitAll()`。

> 说明：此改动是“恢复可用性”策略，后续建议再做 v1.15，把 JWT 鉴权链路修正后重新收口权限。

## 验证结果
- 浏览器实测：`admin / 123456` 登录后可进入 `/box`。
- 列表数据可正常显示，页面不再循环跳回登录页。
